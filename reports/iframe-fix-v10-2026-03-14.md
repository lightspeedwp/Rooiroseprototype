## Problem

```

    at r.cleanup (figma webpack artifacts)
    at s.cleanup (figma webpack artifacts)
    at eI.setupMessageChannel (figma_app)
    at e.onload (figma_app)
```

**Conclusion**: The issue is NOT about timing—it's about **React Router's fundamental incompatibility with Figma's iframe message channel**.

## Solution: v10.0 — Disable Routing in Iframe

### Pragmatic Approach

Since React Router initialization always conflicts with Figma's message port, we eliminate the conflict entirely:

**Strategy**:
- ✅ **Figma Make (iframe)**: Show HomePage directly, NO routing
- ✅ **Production (browser)**: Full React Router with all routes

**Result**: No routing = No message channel conflict = No errors

### Iframe Detection

```tsx
const isInFigmaIframe = (() => {
  try {
    return window.self !== window.top;
  } catch {
    return true; // Security error = iframe
  }
})();
```

**How it works**:
- `window.self !== window.top` — Detects if page is in iframe
- `try/catch` — Handles cross-origin security restrictions
- Returns `true` if in iframe, `false` if standalone

### Conditional Rendering

```tsx

  return <HomePage />; // Single page mode
```

## Root Cause Explanation

### Why All Delays Failed

React Router initialization involves:
1. Creating BrowserRouter instance
2. Setting up history API
3. Attaching window event listeners
4. Managing browser history state

**Conflict**: These operations compete with Figma's iframe message channel setup, regardless of delay timing.

### Why Disabling Routing Works

No router initialization = No event listener conflicts = No message port destruction.

**Simple**: Can't conflict with what doesn't exist.
