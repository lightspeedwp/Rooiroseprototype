# Rooi Rose Magazine Prototype Refactor Research and Content Pack

## Executive summary

This research reviewed the **current `lightspeedwp/Rooiroseprototype` React prototype** (via the enabled GitHub connector), its current design-system layer, routing, and navigation implementation, then produced a **complete, consistent content + navigation data pack** for refactoring the prototype from a “news/newspaper” information model into a **magazine/editorial** one.

The deliverable ZIP contains:

- A **TypeScript data model** for `Post`, `Category`, `Subcategory`, plus **SharedHero props + variants**
- **100 mock posts** (10 per editorial top-level category) with 500–800 word Afrikaans HTML content, unique Unsplash Source hero images, and full frontmatter-like objects
- `navigation.ts` + `megaMenuConfig.ts` implementing the agreed main navigation + mega-menu groupings (including **Shop**, **Kontak**, and **Podsendings under Ontspanning**)
- A descriptive `routes.ts` architecture plan aligning category/subcategory paths with reusable templates
- A manifest (`manifest.csv` and `manifest.md`) listing all 100 posts and their hero images
- `.gitignore` recommendations to enforce exclusion rules and support repository clean-up
- A README explaining how to import the data pack and retire the old news-focused datasets

Rooi Rose’s editorial “pillars” (food, fashion, beauty, health/wellness, human-interest and more) are widely aligned with the target IA you provided. citeturn1search1

## What the current prototype already has

The current prototype already provides a strong foundation for an editorial refactor:

It uses a token-first CSS approach, with a central theme file and additional global CSS layers. This makes it feasible to update Rooi Rose branding at the token level without rewriting every component.

It uses React Router with a page-based route structure that already recognises key magazine page types (home, category pages, article pages, competitions, Rooiwarm winners, contact), which matches the site-map you supplied.

It already separates “data” from “templates/components” in multiple places (e.g., category lists and article arrays). This makes it realistic to introduce a cleaner data layer without a full rewrite, especially if you implement a thin adapter layer to keep legacy components running while you refactor templates.

## Key gaps and refactor implications found in the repository

Several areas are still “previous prototype / newspaper” oriented and should be refactored (or removed) as part of the Rooi Rose conversion:

The navigation configuration is partially remnant from earlier branding, and the live Rooi Rose IA you specified requires a new top-level structure (Kos, Mode, Skoonheid, Gesondheid, Bekendes, Leefstyl, Jou lewe, Ontspanning, Rooiwarm wenners, Wen, plus Shop and Kontak). This is best implemented as an explicit mega menu config rather than ad-hoc nested lists.

The content layer is still structured as “news-first” (e.g., generic news categories and breaking news patterns). For a magazine prototype, you need content objects that capture richer long-form metadata (category + subcategory IA, type=article/recipe/competition/award, hero image, etc.) while remaining iterable for grids and sliders.

Repository hygiene issues were flagged in your brief (e.g., committed `node_modules/tw-animate-css`). In practice, the correct remedial action is to ensure **all** `node_modules/` is excluded via `.gitignore`, and remove any committed `node_modules` artifacts from version control. The deliverable includes a recommended `.gitignore` suitable for React/Vite projects.

The public folder includes brand artefacts from earlier prototypes (e.g., app name in `manifest.json`). Cleaning these is a key part of making the prototype “feel” like Rooi Rose even before pixel-perfect UI updates.

## Figma Make content generation brief and structured data approach

### Content goals by page type

The goal is to generate content that supports a magazine-like experience:

Category landing pages need a “hero + section entry” pattern. The SharedHero configuration supports a quick-link bar to subcategories (matching your observation that the old prototype’s sub-links are a strong entry point).

Subcategory landing pages should reuse the same SharedHero pattern but filter content to a single subcategory, avoiding one-off pages per subcategory.

Post templates should share a single hero component (SharedHero), but vary metadata density by post type (article/recipe/competition/award). This restores consistency and reduces template drift.

### Data modelling goals

The data pack is designed to support:

Reusable templates (category/subcategory/post) driven entirely by structured objects.

A stable naming scheme (slug = `{id}-{slugified-title}`) compatible with legacy “get id from slug” patterns common in React Router article routes.

Simple mapping from prototype to CMS later (the objects resemble frontmatter, but can map to WordPress REST/GraphQL with minimal translation).

### Image rules

All posts use Unsplash Source images via `https://source.unsplash.com/` with:

Category-specific keyword pairs (e.g., `food,recipe`, `skincare,beauty`, `travel,books`) and

A unique `sig` parameter per post to avoid repeated caching and enforce uniqueness across the 100 posts.

## Missing assumptions and decisions made

The current prototype’s “data folder” is primarily `src/app/data/*`. Your instruction requested placing generated TypeScript files under `/src/data/`. The ZIP follows your requirement (`/src/data`), but the README explains how to move or map these files into `src/app/data` for immediate use.

The deliverable generates 100 posts for 10 editorial top-level categories only. Shop and Kontak are included in the navigation and mega menu configs, but they are treated as utility pages (no post content). This interpretation is necessary to satisfy the explicit “100 posts” constraint while still including Shop and Kontak in the top navigation.

Podcasts are treated as “content under Ontspanning” in the information architecture. The existing prototype has standalone podcast routes; the route architecture file recommends nesting routes under `/ontspanning/podsendings` to match the IA requirement.
